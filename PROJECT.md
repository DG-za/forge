# Project-Specific Rules

This file holds project-specific rules, patterns, and conventions that evolved during development. It takes precedence over `AGENTS.md` when they conflict.

`AGENTS.md` is generated from standard templates and can be regenerated by `/scaffold`. Don't put project-specific rules there — they'll be overwritten.

## Development Workflow: Test-Driven Development 🧪

This project follows **strict TDD**. For every feature or fix:

1. **Write tests first.** Read the issue/acceptance criteria. Write tests that describe the expected behaviour. Run them — they should fail.
2. **Implement until tests pass.** Write the minimum code needed to make the tests green.
3. **Refactor.** Clean up with tests still passing.
4. **Commit.** Tests must pass before every commit.

This applies to:
- **All Forge code** (dispatcher, web UI, shared modules) — every PR must include tests written before the implementation.
- **Worker agents dispatched by Forge** — when a worker implements an issue on a target repo, it follows the same loop: write tests from acceptance criteria, then implement.

**No exceptions.** Don't write implementation code before tests exist for the behaviour you're adding. If you're unsure what to test, that means the acceptance criteria are unclear — clarify first.
